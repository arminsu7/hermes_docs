# Qwen3-ASR 在 AGX Orin 上的量化兼容性评估

## 1. 评估结论

对当前这台 AGX Orin 上的 `Qwen3-ASR-1.7B` 来说，量化与推理路线的推荐优先级如下：

1. `bf16` 原模型：最稳，适合先跑通服务并建立基线
2. `AWQ / GPTQ`：如果坚持使用 vLLM 路线，这是最值得优先尝试的量化方向
3. `compressed-tensors INT4`：当前不建议继续作为主部署路线
4. `FP8`：不建议作为这台 Orin 上的主推路线

一句话总结：

对 Jetson AGX Orin 这类边缘设备，主瓶颈不是“理论算力不够”，而是“运行时兼容性、内存带宽和 backend 支持成熟度”。因此量化路线应优先选择生态成熟、加载路径稳定的格式，而不是纸面压缩率最高的格式。

## 2. 已验证的机器与模型信息

以下信息均来自实际检查结果。

### 2.1 系统内存

`free -h` 显示：

- 总内存：61 GiB
- 已用：11 GiB
- 可用：48 GiB
- Swap：30 GiB

### 2.2 GPU 信息

`torch.cuda.get_device_capability(0)` 返回：

```text
(8, 7)
```

即 Jetson AGX Orin 对应的 `SM 8.7`，属于 Ampere 系列，不是 Hopper。

### 2.3 模型目录大小

实际检查模型目录大小：

- `Qwen3-ASR-1.7B`：`4.4G`
- `Qwen3-ASR-1.7B-Int4`：`2.1G`

这说明 INT4 量化后模型体积明显下降，但体积下降不等于部署链路就兼容。

### 2.4 模型结构信息

从 `config.json` 读取到的关键结构参数如下：

- `text hidden_size = 2048`
- `text num_hidden_layers = 28`
- `text num_attention_heads = 16`
- `audio encoder_layers = 24`
- `audio d_model = 1024`

这说明该模型并非纯文本 decoder-only LLM，而是“音频编码器 + 文本解码器”的 ASR 结构。相比普通文本模型，其 backend 兼容性更依赖自定义实现是否完善。

### 2.5 当前 vLLM 运行时

容器内实际版本为：

- `vllm 0.14.0`
- `torch 2.9.1`

### 2.6 当前 vLLM 已注册的量化方法

在该环境中，vLLM 实际注册的量化方法列表包含：

- `awq`
- `gptq`
- `awq_marlin`
- `gptq_marlin`
- `fp8`
- `compressed-tensors`
- `gguf`
- `bitsandbytes`
- 等

重要说明：

“vLLM 注册了某种量化方法”只表示框架层面识别该格式，不代表当前 `Qwen3-ASR` 这套自定义 backend 在 AGX Orin 上也完全打通了该格式的加载、参数映射和 kernel 路径。

## 3. 主瓶颈分析

### 3.1 运行时兼容性是第一瓶颈

当前最直接的问题不是模型太大，而是量化格式无法被当前后端正确加载。

前一份分析已经证明：
- `compressed-tensors INT4` 在当前 `qwen_asr + vLLM` 组合下会在权重加载阶段报错
- 报错点是 packed weight 的参数名映射不兼容

因此，当前第一瓶颈不是容量，而是“格式和加载器是否匹配”。

### 3.2 内存带宽与 unified memory 是第二瓶颈

Jetson AGX Orin 不是 H100/A100 这类独立 HBM 大卡，而是 unified memory 架构。即使量化后权重变小，推理性能仍强依赖：

- 权重访问模式
- KV cache 读写
- packed kernel 是否高效
- backend 是否触发最优实现路径

换句话说，权重压成 INT4 只是“减少运货量”，并不保证“收费站更快”。如果 backend 不顺，省下来的体积优势未必能兑现为稳定吞吐。

### 3.3 kernel / backend 成熟度是第三瓶颈

Orin 的 `SM 8.7` 在量化 kernel 生态里并不总是第一优先级。常见情况是：

- 文档说支持某量化方式
- 但最优 kernel 可能优先覆盖 `sm80 / sm86 / sm89 / sm90`
- `sm87` 有时会退化到泛化实现或缺少最佳路径

因此，在 Orin 上做量化部署时，“支持”与“好用”是两回事。工程上更应优先选成熟度高的路径。

## 4. 各量化路线评估

## 4.1 bf16 原模型

### 推荐级别

首选基线。

### 为什么推荐

1. 当前原模型目录只有 `4.4G`，远未到完全装不下的程度
2. 机器可用内存约 `48GiB`，对 1.7B 级 ASR 模型较为宽松
3. 可以先把“服务逻辑问题”和“量化兼容性问题”解耦

### 适用目标

- 先把服务稳定跑起来
- 验证流式识别链路
- 建立延迟、占用和识别质量基线

### 为什么这是优先选项

当前如果一上来就用量化模型，会把两类问题耦合在一起：

1. ASR 服务业务逻辑是否正常
2. 量化权重是否能被 backend 正确消费

先用 `bf16` 跑通，才能把后续量化评估建立在稳定基线上。

## 4.2 AWQ

### 推荐级别

量化首选。

### 为什么优先于 compressed-tensors

1. 在 vLLM 生态中，AWQ 属于更成熟的部署路线
2. 与 `compressed-tensors + 自定义 ASR backend` 相比，参数映射和加载路径通常更直接
3. 如果你坚持继续使用 vLLM，AWQ 的工程风险更低

### 适用目标

- 希望保留 vLLM 路线
- 希望把显存/内存占用压下去
- 希望避免再次踩到 packed weight 协议不匹配的问题

### 风险提示

这并不意味着 AWQ 对 Qwen3-ASR 一定零成本可用，而是相对于当前已知失败的 `compressed-tensors INT4`，它更值得优先验证。

## 4.3 GPTQ

### 推荐级别

量化次选，紧随 AWQ。

### 为什么仍然值得考虑

1. GPTQ 在 vLLM 中也是成熟量化路线之一
2. 它通常更接近“标准部署格式”，而非需要深度配合 loader 的工程格式
3. 作为 INT4 权重量化方案，它保留了较好的压缩收益

### 与 AWQ 的关系

如果要在 `AWQ` 和 `GPTQ` 之间排序，当前更建议：

```text
AWQ > GPTQ
```

原因不是 GPTQ 不好，而是当前场景下更优先追求“部署稳健性和少踩坑”。如果已有成熟 GPTQ 工具链，也可以反过来先试 GPTQ。

## 4.4 compressed-tensors INT4

### 推荐级别

当前不建议继续作为主部署路线。

### 现象

- 目录大小仅 `2.1G`
- 压缩率看起来很有吸引力

### 为什么不建议主推

1. 已实际验证在当前 `qwen_asr + vLLM` 组合下启动失败
2. 问题不是调参数可绕过的小故障，而是 backend 与量化权重协议未打通
3. 继续使用它需要把这条路线视为“研发适配任务”，而不是“交付可用方案”

### 为什么别人文档里看起来能用

因为很多 `compressed-tensors` 示例针对的是标准 decoder-only 文本 LLM，而不是当前这种：

- ASR 结构
- 自定义 `qwen_asr` backend
- Jetson Orin 环境

这些附加条件叠在一起，兼容风险明显增大。

## 4.5 FP8

### 推荐级别

当前不建议作为主推路线。

### 原因一：硬件代际不匹配

Jetson AGX Orin 是 Ampere `SM 8.7`，不是 Hopper。FP8 真正最成熟、最自然的高性能部署土壤，通常是 H100/H200 这一代。

### 原因二：软件收益不确定

当前主要矛盾是 backend 兼容性，而不是简单的权重精度问题。此时切换到 FP8，并不是在解决主矛盾，而是在引入一条新的实验路线。

### 原因三：模型类型限制

当前模型不是普通文本模型，而是 Qwen3-ASR 自定义结构。即使框架层面支持 FP8，也不等于这套自定义模型链路对 FP8 的加载、推理和精度控制都成熟。

## 5. 为什么这里不优先推荐 GGUF / llama.cpp

对于普通文本 LLM，在 Jetson 上常常会优先考虑 `GGUF + llama.cpp`，因为这条路线简单、鲁棒、生态成熟。

但当前任务是 `Qwen3-ASR`，不是纯文本聊天模型。它具备以下特征：

- 有音频编码器
- 有 ASR 专用前后处理
- 有流式状态机
- 当前工程已经围绕 `qwen_asr + vLLM / transformers` 构建

因此，这里不优先推荐 GGUF，并不是 GGUF 不好，而是它并不一定适合这套 ASR 结构和现有工程栈。

## 6. 实际推荐排序

如果目标是“尽快得到一个能上线测试的方案”，推荐排序如下：

### 路线 1：先用 bf16 原模型

用途：
- 跑通服务
- 建立识别质量基线
- 建立延迟和占用基线

### 路线 2：如果必须压内存，优先尝试 AWQ

用途：
- 保留 vLLM 生态
- 获取较好的权重压缩收益
- 尽量降低 backend 协议不兼容风险

### 路线 3：AWQ 不顺时再试 GPTQ

用途：
- 作为第二条成熟的 INT4 权重量化路线
- 保留部署生态一致性

### 路线 4：将 compressed-tensors 降级为研发分支

用途：
- 作为后续 backend 适配任务
- 不作为当前上线主路径

### 路线 5：FP8 暂不优先

用途：
- 仅在明确要做 FP8 实验时再考虑
- 不建议作为当前 Orin 主部署路线

## 7. 按目标场景给建议

### 7.1 如果目标是“先可用”

推荐：

```text
bf16 原模型
```

### 7.2 如果目标是“继续用 vLLM 且压缩内存”

推荐：

```text
AWQ > GPTQ > compressed-tensors
```

### 7.3 如果目标是“研究 llmcompressor 量化产物如何部署”

推荐：

把 `compressed-tensors` 视为一个需要专项 backend 适配的研发任务，而不是现成可交付方案。

### 7.4 如果目标是“追求极限性能”

当前不建议先追新格式，而应优先做好基础调优：

- backend 路线稳定性
- streaming chunk 配置
- batch size / concurrency
- `max_new_tokens`
- 内存占用与上下文长度平衡

这些调整的收益通常比盲目换量化格式更稳定。

## 8. 最终结论

对 AGX Orin 上的 `Qwen3-ASR-1.7B` 而言，当前最合适的路线不是继续死磕 `llmcompressor compressed-tensors INT4`，而是：

首选：
- `bf16` 原模型作为稳定基线

量化首选：
- `AWQ`

量化备选：
- `GPTQ`

当前不建议：
- `compressed-tensors` 作为 vLLM 主部署格式
- `FP8` 作为 Orin 主推路线

一句话说透：

当前这台 Orin 的主矛盾不是“模型不够小”，而是“量化格式是否真的能被这条 ASR 推理链路稳定加载并正确执行”。因此选型优先级应让位于兼容性与成熟度，而不是只看压缩率。
